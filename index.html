Por qué un proyecto de fútbol

Cuando empecé a explorar Microsoft Fabric me encontré con el problema clásico de aprender una tecnología nueva: la documentación oficial explica muy bien las piezas por separado, pero cuesta ver cómo encajan entre sí hasta que no las usas en algo concreto.

Decidí construir un proyecto real desde cero. Elegí el fútbol como caso de uso porque es fácil de entender para cualquiera: hay partidos, hay goles, hay una clasificación que cambia en función de los resultados. Eso encaja a la perfección con lo que hace el streaming de datos, que es exactamente procesar información que llega de forma continua y en tiempo real.

"La mejor forma de aprender una tecnología nueva es construir algo con ella, aunque sea con datos inventados."

El resultado es un sistema donde puedes ver en directo cómo avanza la clasificación de la liga, quién lidera la tabla de goleadores y qué equipos acumulan más tarjetas, todo actualizado automáticamente cada 30 segundos mientras el simulador está generando partidos.

Qué es Microsoft Fabric y por qué importa

Si llevas tiempo en el mundo de los datos habrás oído hablar de servicios como Azure Event Hubs, Azure Stream Analytics, Azure Data Explorer o Power BI. Son herramientas potentes, pero usarlas juntas implica configurar conexiones entre ellas, gestionar credenciales independientes, pagar por cada servicio por separado y tener conocimientos de infraestructura cloud que no siempre se tienen cuando estás aprendiendo.

Microsoft Fabric es la respuesta de Microsoft a ese problema. Es una plataforma unificada que integra todos esos componentes en un solo sitio, con una interfaz web accesible y sin necesidad de tocar Azure directamente. Todo el proyecto vive en el mismo workspace.

Lo que utilizamos en concreto es la capacidad de Real-Time Intelligence de Fabric, el conjunto de herramientas diseñado para trabajar con datos que llegan de forma continua: ingesta en streaming, almacenamiento analítico de baja latencia y visualización en tiempo real.

La arquitectura del proyecto

Antes de construir nada, conviene entender cómo fluyen los datos de principio a fin. El pipeline tiene cinco componentes encadenados:

El script de Python genera los eventos de cada partido y los envía al Event Hub. El Eventstream los recoge y transporta al Eventhouse con Direct Ingestion. El Eventhouse los almacena en una KQL Database con vistas materializadas que mantienen la clasificación siempre calculada. El Dashboard consulta esas vistas cada 30 segundos y actualiza los tiles.

Eventstream

El mensajero del pipeline. Recibe los eventos y los enruta al destino sin almacenar nada. Gestiona la conexion entre la fuente Python y la base de datos KQL.

Eventhouse

El almacen analitico. Guarda todos los eventos con indexacion optimizada para consultas en tiempo real. Aqui viven los goles, tarjetas y la clasificacion.

Paso 1

Montar la infraestructura en Fabric

Todo se hace desde la interfaz web de Fabric. No hay que tocar Azure ni escribir codigo de infraestructura. Son cuatro pasos consecutivos:

  • 1
    Crear el Workspace

    El contenedor de todos los recursos del proyecto. Equivale a una carpeta de trabajo en la nube. Usamos la licencia Trial gratuita de 60 dias.

  • 2
    Crear el Eventhouse

    La base de datos KQL. Al crearlo, Fabric genera automaticamente la base de datos interior. Activamos el Streaming Ingestion para que acepte datos en tiempo real.

  • 3
    Crear el Eventstream con fuente Custom endpoint

    El transportista de datos. Al crear la fuente, Fabric nos proporciona una Connection String que usaremos en el script Python para enviar eventos.

  • 4
    Conectar Eventstream con el Eventhouse

    Configuramos el destino con Direct Ingestion, formato JSON y tabla EventosPartido. Esto garantiza que los eventos llegan a la base de datos en menos de un segundo.

Paso 2

El simulador de partidos en Python

El simulador es un Notebook de Python que vive dentro del propio Fabric. No hace falta instalar nada en el ordenador — todo corre en la nube sobre Apache Spark, que es el motor de procesamiento que usa Fabric internamente para ejecutar codigo.

El simulador genera 20 equipos con nombres reales de la liga espanola, 300 jugadores sinteticos distribuidos por posicion y dorsal, y simula una jornada completa de 10 partidos. Por cada accion que ocurre genera un mensaje JSON y lo envia al Event Hub.

Una decision interesante del diseno es la fuerza por equipo: cada club tiene un valor entre 0 y 1 que afecta a la probabilidad de que marque en cada accion. Esto permite controlar los resultados de la simulacion, por ejemplo dando al Atletico de Madrid muchas mas opciones de ganar que al resto.

Hay cinco tipos de evento que el simulador puede generar:

Cada evento viaja como un objeto JSON con toda la informacion relevante: el nombre del jugador, el equipo, el minuto, el marcador en ese momento. El nombre del jugador viaja dentro del evento, por lo que cuando llega al Eventhouse ya esta disponible para mostrarlo en el dashboard sin necesidad de cruzar tablas adicionales.

Paso 3

KQL: consultas para datos en movimiento

Una vez que los datos estan en el Eventhouse los consultamos con KQL, Kusto Query Language. Si tienes experiencia con SQL la logica es muy parecida, pero la sintaxis cambia: en KQL las consultas se construyen como una cadena de transformaciones encadenadas con el operador |, lo que las hace muy legibles una vez que te acostumbras.

Por que KQL y no SQL

Aspecto SQL KQL
Disenado para Datos relacionales estaticos Series temporales y streaming
Sintaxis SELECT ... FROM ... WHERE Pipeline con operador |
Rendimiento en streaming No optimizado de serie Nativo, indexado por tiempo
Vistas materializadas Soporte limitado Primera clase, automaticas
Mejor para ERP, CRM, transaccional Logs, IoT, telemetria, eventos

Las vistas materializadas

Este es el concepto mas importante del proyecto desde el punto de vista del rendimiento. Imagina que tras varias jornadas acumulas 100.000 eventos en la tabla principal. Si cada vez que el dashboard se refresca tuvieramos que recorrer todos esos registros para calcular quien es el maximo goleador, las consultas serian lentas e ineficientes.

Las vistas materializadas resuelven esto: son tablas de resumen que Fabric mantiene actualizadas de forma automatica con cada evento nuevo. Cuando llega un gol, la vista GoleadoresAcum se actualiza al instante. El dashboard consulta esa tabla resumida, no los 100.000 eventos originales.

Creamos tres vistas materializadas: GoleadoresAcum, AsistenciasAcum y TarjetasAcum. Todas se crean con el parametro backfill=true, que indica a Fabric que al crear la vista procese tambien todos los datos historicos que ya existian en la tabla, no solo los nuevos.

Paso 4

El Real-Time Dashboard

El ultimo paso es el Real-Time Dashboard de Fabric. A diferencia de Power BI, orientado al analisis historico, este tipo de dashboard esta disenado para actualizar sus tiles de forma continua consultando directamente la KQL Database.

Configuramos el auto-refresco a 30 segundos y creamos 12 tiles con distintos tipos de visualizacion. La combinacion de tablas con rankings, graficos de lineas para ver la evolucion de los equipos a lo largo de las jornadas, graficos de barras para comparar estadisticas por club y tiles de numero grande para los totales de la jornada da una vista completa del estado de la competicion.

Tiles de tabla

Clasificacion, goleadores, asistencias, partidos en vivo, ultimos goles y tarjetas. Se actualizan con los datos mas recientes del Eventhouse.

Tiles de grafico

Evolucion Top 5 en grafico de lineas, goles por equipo y por tramo de minuto en barras, distribucion de tipos de evento en grafico de sectores.

Con el dashboard abierto en una pestana y el Notebook ejecutandose en otra, los tiles cambian visiblemente cada 30 segundos. La clasificacion se reordena segun los resultados y los goleadores escalan posiciones en tiempo real. Es el momento mas visual del proyecto.

Por que merece la pena aprender Microsoft Fabric

Esta es la pregunta mas habitual cuando cuento este proyecto. Mi respuesta: porque permite construir cosas que antes requeian mucho mas conocimiento de infraestructura y varios servicios coordinados.

  • 1
    Todo en un workspace

    Sin saltar entre servicios de Azure ni configurar conexiones entre componentes. El Notebook, el Eventstream, el Eventhouse y el Dashboard conviven en el mismo lugar. El tiempo se invierte en los datos, no en la infraestructura.

  • 2
    Latencia inferior a un segundo

    Con Direct Ingestion los eventos pasan de Python al Eventhouse casi de forma inmediata. En una arquitectura Azure tradicional esto exigiria coordinar varios servicios con sus propias configuraciones y costes.

  • 3
    Trial gratuito de 60 dias

    Suficiente para aprender, construir un proyecto completo y tenerlo como referencia en el portfolio. No requiere suscripcion de Azure ni tarjeta de credito.

  • 4
    Arquitectura escalable sin cambios

    Si manana se quisiera conectar una API de datos de futbol real en lugar del simulador, solo cambiaria el Notebook. El Eventstream, el Eventhouse y el Dashboard seguirian funcionando igual sin modificaciones.

Tecnologias utilizadas

El proyecto se construyo enteramente dentro de Microsoft Fabric, usando los siguientes componentes: Notebook de Python con PySpark como simulador, Eventstream para la ingesta en streaming, Eventhouse con KQL Database para el almacenamiento analitico, y Real-Time Dashboard para la visualizacion. Las consultas y vistas materializadas se escribieron en KQL.

Para terminar

Este proyecto demostro que con Microsoft Fabric es posible construir un pipeline completo de datos en tiempo real, desde la generacion de eventos hasta la visualizacion, sin necesidad de gestionar infraestructura externa ni combinar multiples servicios de Azure por cuenta propia.

Lo mas valioso del proceso no fue solo el resultado final, sino todo lo que queda claro por el camino: como funciona el streaming de datos, por que KQL es la herramienta correcta para este tipo de casos, que son las vistas materializadas y como Fabric integra todas las piezas en una sola plataforma coherente.

Si estas empezando en el mundo de los datos y quieres aprender herramientas con alta demanda en el mercado, Fabric es una apuesta solida. El trial gratuito da tiempo mas que suficiente para construir algo como esto de principio a fin y entender de verdad como funciona el dato en tiempo real.
